# SuperDev Studio 工作台改版发布说明

## 1. 概览

本次改版将 SuperDev Studio 从“功能模块集合”重构为“以 change 驱动的交付工作台”。

核心目标只有一个：

> 让团队围绕一次明确的改版批次（ChangeBatch）持续推进，而不是在项目、任务、流水线、记忆、知识等多个孤立页面之间手工切换。

## 2. 为什么要改

旧版设计存在 5 个主要问题：

1. **信息架构偏模块导向**
   - 用户目标是推进一次交付，而不是逐个访问模块页。

2. **执行入口重复**
   - 项目推进与流水线启动语义接近，容易造成执行链路分叉。

3. **任务与真实 change 流程脱节**
   - 任务板适合规划，但不应该单独承担交付事实来源。

4. **上下文能力分散**
   - 记忆、知识、上下文打包之间存在天然耦合，却分散在不同页面。

5. **运行结果不可稳定追踪**
   - 缺少批次 ID 与外部 `change_id` 时，难以复盘一次运行究竟属于哪轮改版。

## 3. 本次改了什么

### 3.1 信息架构

一级导航从旧版的功能页集合，调整为：

- 工作台
- 工作区
- 变更中心
- 交付运行
- 上下文中心
- 项目设置

新的结构把“项目默认配置 → 变更批次 → 运行执行 → 上下文沉淀”串成一条主线。

### 3.2 数据模型

新增与增强的关键对象如下：

- `Project`
  - 新增默认平台、前后端、领域、上下文模式、Token 预算、动态召回、记忆回写等字段
- `ChangeBatch`
  - 新增本轮改版执行单元，承载 `title/goal/status/mode/external_change_id/latest_run_id`
- `PipelineRun`
  - 新增 `change_batch_id` 与 `external_change_id`，用于串联真实执行链路

### 3.3 页面体验

- **工作台**：展示工作区指标和最近交付运行
- **工作区**：保留项目与任务规划能力，但弱化其对真实交付状态的主导地位
- **变更中心**：创建并选中当前 `ChangeBatch`
- **交付运行**：运行表单自动继承项目默认配置，并绑定当前批次
- **上下文中心**：统一承载 Context Pack / 记忆库 / 知识库
- **项目设置**：集中维护默认执行画像

## 4. 对团队的直接收益

1. **交付入口更明确**
   - 先选工作区，再选批次，再启动交付。

2. **配置重复录入更少**
   - 默认技术栈和上下文策略沉到项目级，减少每次运行的重复填写。

3. **运行复盘更稳定**
   - 可以追踪某次运行属于哪个 `ChangeBatch`、对应哪个外部 `change_id`。

4. **上下文沉淀更自然**
   - 记忆与知识不再只是孤立资料，而是为后续运行服务的上下文资产。

## 5. 兼容性说明

- `/memory` 与 `/knowledge` 旧路由目前仍保留兼容入口
- 旧的项目与任务能力仍可使用，但推荐从“变更中心 → 交付运行”发起真实交付
- 旧仓库中与当前后端重复的根目录部署脚手架已移除，避免与现有 `backend/` 结构冲突

## 7. 风险与后续计划

以下事项不阻断本轮发布，但建议作为下一轮工程化改进继续推进：

1. 补齐根级 CI 主流程
2. 实施前端路由级懒加载与包体优化
3. 将 `/memory` 与 `/knowledge` 完全收敛到 `/context`
4. 继续拆分后端超大文件，降低维护成本

## 8. 推荐给评审 / PR 的摘要版本

可以直接使用下面这段作为 PR 描述：

> 本次改版将 SuperDev Studio 重构为 change 驱动的交付工作台，新增 ChangeBatch 作为执行单元，补齐 Project 默认执行画像与 PipelineRun 追踪元数据，并将前端导航重构为工作台、工作区、变更中心、交付运行、上下文中心、项目设置六大入口。这样可以减少重复执行入口、降低配置重复填写、增强运行与产物的可追踪性，同时为后续以 super-dev 持续推进改版提供稳定的 UI 与数据基础。
